# Tweet Scheduler

A tweet scheduling tool with a static frontend on GitHub Pages and a serverless Google Apps Script backend.

| Layer | Technology | Cost |
|-------|-----------|------|
| Frontend | HTML5 + Tailwind CSS v3 + Vanilla JS | Free (GitHub Pages) |
| Backend | Google Apps Script (`doPost` JSON API) | Free (GAS quota) |
| Database | Google Sheets | Free |
| CI/CD | GitHub Actions | Free |

## How It Works

```
User (browser)
     │  opens GitHub Pages URL
     ▼
frontend/public/index.html
     │  fetch() POST with JSON body
     ▼
Google Apps Script Web App (doPost)
     │  reads/writes
     ▼
Google Sheets ("tweet" sheet)
     │  scheduler trigger runs every minute
     ▼
Twitter/X API v2 (posts tweets)
```

1. The user opens the static site hosted on GitHub Pages.
2. On form submission, the browser calls the GAS web app URL via `fetch()`.
3. GAS validates the input, writes a row to Google Sheets, and either posts immediately or leaves it for the scheduler.
4. A time-based GAS trigger runs `runScheduler()` every minute, evaluating cron expressions and posting tweets when they match.

## Sheet Schema

The GAS backend reads/writes a Google Sheet named **`tweet`** with exactly these columns:

| Column | Header | Description |
|--------|--------|-------------|
| A | `tweet link` | Source tweet URL (twitter.com or x.com), or blank for new tweets |
| B | `resource links` | Comma-separated media URLs, `none`, or `error: ...` |
| C | `status` | Blank (pending), `sent`, or `error: ...` |
| D | `title` | Tweet text to post |
| E | `cron expression` | 5-field cron string for scheduled posting, or blank |
| F | `max count` | Maximum number of times to post (0 = unlimited) |
| G | `post count` | Number of times posted so far |

Before you start, make sure you have:

- A **Twitter/X developer account** with an app that has Read and Write permissions

- **Node.js ≥ 18** installed locally (only needed for local development and running tests)

## Part 1 — Backend Deployment (Google Apps Script)

The backend is a set of `.gs` files deployed as a GAS web app. It has no hosting costs — Google runs it on their infrastructure.

### Step 1 — Create the Google Sheet

1. Go to [sheets.google.com](https://sheets.google.com) and create a new blank spreadsheet.
2. Give it any name you like (e.g. **Tweet Scheduler**).
3. You do not need to create the `tweet` tab manually — the script creates it automatically on first run.
4. Note your spreadsheet URL. It looks like:
   ```
   https://docs.google.com/spreadsheets/d/SPREADSHEET_ID/edit
   ```
   Keep the `SPREADSHEET_ID` — you may need it if you use a standalone GAS project.

---

### Step 2 — Open the Apps Script editor

The recommended approach is a **spreadsheet-bound** project (the script runs in the context of your sheet automatically).

1. Open your Google Sheet.
2. Click **Extensions → Apps Script**.
3. A new Apps Script project opens. It is already bound to your spreadsheet — `SpreadsheetApp.getActiveSpreadsheet()` will work without any extra configuration.
4. You will see a default file called `Code.gs`. You can rename or delete it — all the code goes into the files you create in the next step.

> **Alternative — Standalone project:** If you prefer a standalone project, go to [script.google.com](https://script.google.com) and click **New project**. Then open `SheetUtils.gs` and replace `SpreadsheetApp.getActiveSpreadsheet()` with `SpreadsheetApp.openById('YOUR_SPREADSHEET_ID')`.

---

### Step 3 — Create script files

You need to create one file in the GAS editor for each `.gs` file in the `scripts/` directory of this repo.

**For each of the following files**, do this:
1. In the Apps Script editor left sidebar, click the **+** next to "Files".
2. Choose **Script**.
3. Name it exactly as shown (without the `.gs` extension).
4. Paste the full contents of the corresponding file from `scripts/`.

Create these 10 files in this order:

| File to create (name in GAS) | Source file in repo |
|------------------------------|---------------------|
| `Constants` | `scripts/Constants.gs` |
| `SheetUtils` | `scripts/SheetUtils.gs` |
| `Extractor` | `scripts/Extractor.gs` |
| `Scheduler` | `scripts/Scheduler.gs` |
| `Poster` | `scripts/Poster.gs` |
| `TwitterClient` | `scripts/TwitterClient.gs` |
| `TriggerManager` | `scripts/TriggerManager.gs` |
| `Main` | `scripts/Main.gs` |
| `WebApp` | `scripts/WebApp.gs` |

> All `.gs` files share a single global scope in GAS. The order does not matter for execution, but creating `Constants` first makes it easier to read the others.

> **Using clasp (optional — advanced):** If you have [clasp](https://github.com/google/clasp) installed, you can push the files directly from the command line:
> ```bash
> npm install -g @google/clasp
> clasp login
> # From the scripts/ directory:
> clasp clone <SCRIPT_ID>     # get Script ID from Apps Script → Project Settings
> cp ../scripts/*.gs .
> cp ../scripts/appsscript.json .
> clasp push
> ```

### Step 4 — Replace the manifest

The `appsscript.json` manifest defines the required OAuth scopes and runtime version.

1. In the Apps Script editor, click **Project Settings** (gear icon ⚙️) in the left sidebar.
2. Check the box **"Show 'appsscript.json' manifest file in editor"**.
3. Click `appsscript.json` in the file list.
4. Replace its entire contents with the contents of `scripts/appsscript.json` from this repo:

```json
{
  "timeZone": "America/New_York",
  "dependencies": {},
  "exceptionLogging": "STACKDRIVER",
  "runtimeVersion": "V8",
  "oauthScopes": [
    "https://www.googleapis.com/auth/spreadsheets",
    "https://www.googleapis.com/auth/script.external_request",
    "https://www.googleapis.com/auth/script.scriptapp",
    "https://www.googleapis.com/auth/script.webapp.deploy"
  ]
}
```

> Change `"timeZone"` to your local timezone if needed (e.g. `"Asia/Kolkata"`, `"Europe/London"`). This affects how cron schedule times are interpreted.

---

### Step 5 — Store Twitter API credentials

The script reads credentials from **Script Properties** at runtime. They are never stored in source code.

1. In the Apps Script editor, click **Project Settings** (gear icon ⚙️).
2. Scroll down to the **Script Properties** section.
3. Click **Add script property** and add all four of the following:

| Property Key | Value |
|---|---|
| `TWITTER_API_KEY` | Your OAuth Consumer Key (API Key) |
| `TWITTER_API_SECRET` | Your OAuth Consumer Secret (API Key Secret) |
| `TWITTER_ACCESS_TOKEN` | Your OAuth Access Token |
| `TWITTER_ACCESS_TOKEN_SECRET` | Your OAuth Access Token Secret |

4. Click **Save script properties**.

> **Security note:** Script Properties are stored encrypted by Google and are never visible in source code or version history. Never paste credentials directly into `.gs` files.

> **Where to get these:** Go to [developer.twitter.com](https://developer.twitter.com) → your project → your app → **Keys and Tokens** tab. Generate or regenerate all four values. Make sure the app has **Read and Write** permissions under **User authentication settings**.

---

### Step 6 — Run initial setup and verify

Before deploying, verify that the script can reach your spreadsheet and that the API credentials are working.

**6a. Authorize the script**

1. In the Apps Script editor, select `getOrCreateTweetSheet` from the function dropdown at the top toolbar.
2. Click **Run**.
3. A dialog will appear: **"Authorization required"** → click **Review permissions**.
4. Choose your Google account.
5. Click **Advanced** → **Go to [your project name] (unsafe)** → **Allow**.
   - This is expected for any new Apps Script project. Google shows this warning for unverified apps.
6. After authorization, the function runs. Check your spreadsheet — a `tweet` tab should appear with 7 column headers.

**6b. Verify credentials and API access**

1. In the function dropdown, select `diagnoseCreds`.
2. Click **Run**.
3. Click **Execution log** (at the bottom of the screen) to see the output.
4. You should see lines like:
   ```
   OK: TWITTER_API_KEY = abc123...
   OK: TWITTER_API_SECRET = xyz789...
   OK: TWITTER_ACCESS_TOKEN = 123456...
   OK: TWITTER_ACCESS_TOKEN_SECRET = 789abc...
   OK: Sheet found/created: tweet
   OK: API call succeeded. Tweet text: just setting up my twttr
   OK: OAuth header built: OAuth oauth_consumer_key="abc1...
   ```
5. If any line says `MISSING` or `ERROR`, fix it before proceeding.

---

### Step 7 — Deploy the web app and copy the URL

This creates the HTTP endpoint that the frontend calls.

1. In the Apps Script editor, click **Deploy** (top right) → **New deployment**.
2. Click the **gear icon ⚙️** next to "Select type" and choose **Web app**.
3. Fill in the settings:
   - **Description:** `Tweet Scheduler v1` (or any label you like)
   - **Execute as:** `Me` (the script runs with your Google account's credentials)
   - **Who has access:** `Anyone` (required so the frontend can call it without authentication)
4. Click **Deploy**.
5. Google will prompt for authorization again — click through and allow all permissions.
6. After deployment, you will see a screen like this:
   ```
   Web app URL
   https://script.google.com/macros/s/AKfycb.../exec
   ```
7. **Copy this URL.** This is your `GAS_URL`. You will need it in Step 10.

> **Important:** The URL ends in `/exec`. Do not use the `/dev` URL for production — `/dev` requires authentication and will not work from the static frontend.

> **Updating after code changes:** Every time you edit a `.gs` file, you must create a new deployment for the changes to take effect. Go to **Deploy → Manage deployments**, click the pencil ✏️ icon on your existing deployment, set Version to **New version**, and click **Deploy**. The URL stays the same.

---

### Step 8 — Enable the scheduler trigger

The scheduler posts cron-scheduled tweets by running every minute in the background.

1. In the Apps Script editor, select `setupTriggers` from the function dropdown.
2. Click **Run**.
3. To verify: click **Triggers** (clock icon ⏰) in the left sidebar.
4. You should see a trigger: `runScheduler` · Time-driven · Every minute.

> **Quota note:** Running every minute uses ~1,440 GAS executions per day. The free tier allows 6 minutes of daily execution time and 20,000 UrlFetch calls — more than enough for typical personal use.

> **To disable scheduling at any time:** Select `removeTriggers` from the dropdown and click Run, or delete the trigger manually from the Triggers view.

### Step 10 — Add the GAS_URL secret

The GitHub Actions workflow replaces `__GAS_URL__` in `frontend/public/js/api.js` with the real GAS URL at build time. The URL is stored as a GitHub Secret so it is never committed to the repo.

1. On GitHub, go to your repository.
2. Click **Settings** (top navigation bar).
3. In the left sidebar, click **Secrets and variables → Actions**.
4. Click **New repository secret**.
5. Fill in:
   - **Name:** `GAS_URL`
   - **Secret:** Paste the full GAS web app URL you copied in Step 7
     (e.g. `https://script.google.com/macros/s/AKfycb.../exec`)
6. Click **Add secret**.

> The secret is now available to GitHub Actions as `${{ secrets.GAS_URL }}`. It is never logged or exposed in workflow output.

---

### Step 11 — Enable GitHub Pages

1. On GitHub, go to your repository → **Settings**.
2. In the left sidebar, click **Pages**.
3. Under **Build and deployment**, set:
   - **Source:** `Deploy from a branch`
   - **Branch:** `gh-pages` · `/ (root)`
4. Click **Save**.

> The `gh-pages` branch does not exist yet — it will be created automatically by GitHub Actions on the first deploy (Step 12).

### Step 12 — Trigger the first deploy

The deploy workflow runs automatically on every push to `main`. If you just pushed in Step 9, it may already be running.

**To check or trigger manually:**

1. On GitHub, go to your repository → **Actions** tab.
2. You should see a workflow run called **"Build & Deploy to GitHub Pages"**.
3. Click on it to watch the live progress. It runs these steps:
   - **Checkout** — clones the repo
   - **Set up Node.js** — installs Node 20
   - **Install dependencies** — runs `npm ci` in `frontend/`
   - **Build CSS** — runs Tailwind, generates `public/css/app.css`
   - **Inject GAS URL** — replaces `__GAS_URL__` in `public/js/api.js` with your secret
   - **Deploy to GitHub Pages** — pushes `frontend/public/` to the `gh-pages` branch
4. The workflow should complete in under 2 minutes. A green ✓ means success.

**To trigger a deploy manually** (without pushing a commit):

1. Go to **Actions → Build & Deploy to GitHub Pages**.
2. Click **Run workflow** → **Run workflow**.

## Cron Expression Reference

The scheduler uses standard 5-field cron syntax. Times are interpreted in the timezone set in `appsscript.json`.

```
┌───────────── minute       (0–59)
│ ┌─────────── hour          (0–23)
│ │ ┌───────── day of month  (1–31)
│ │ │ ┌─────── month         (1–12)
│ │ │ │ ┌───── day of week   (0–6, Sunday = 0)
│ │ │ │ │
* * * * *
```

| Expression | Meaning |
|---|---|
| `* * * * *` | Every minute |
| `0 9 * * *` | Every day at 9:00 AM |
| `0 9 * * 1` | Every Monday at 9:00 AM |
| `30 18 * * 5` | Every Friday at 6:30 PM |
| `0 12 1 * *` | 1st of every month at noon |
| `*/5 * * * *` | Every 5 minutes |
| `0 9,17 * * 1-5` | Weekdays at 9 AM and 5 PM |
| `0 0 * * 0` | Every Sunday at midnight |

**Supported syntax per field:**

| Syntax | Example | Meaning |
|--------|---------|---------|
| `*` | `*` | Any value |
| Specific number | `9` | Exactly 9 |
| Range | `1-5` | 1 through 5 |
| Step | `*/15` | Every 15 units |
| List | `1,3,5` | 1, 3, or 5 |
| Range + step | `1-5/2` | 1, 3, 5 |

## Troubleshooting

### "Missing Twitter API credentials in PropertiesService"
Script Properties are not set. Follow [Step 5](#step-5--store-twitter-api-credentials). Make sure all four keys are present: `TWITTER_API_KEY`, `TWITTER_API_SECRET`, `TWITTER_ACCESS_TOKEN`, `TWITTER_ACCESS_TOKEN_SECRET`.

### "Tweet link must be a valid twitter.com or x.com status URL."
The URL must match `https://twitter.com/{user}/status/{id}` or `https://x.com/{user}/status/{id}`. Profile URLs, search URLs, and shortened links (like `t.co`) are not supported.

### "error: HTTP 401" in the resource links or status column
Your API credentials are invalid or expired. Go to [developer.twitter.com](https://developer.twitter.com) → your app → **Keys and Tokens** → regenerate the Access Token and Access Token Secret → update Script Properties.

### "error: HTTP 403" when posting
Your app does not have Write permissions. Go to [developer.twitter.com](https://developer.twitter.com) → your app → **User authentication settings** → set App permissions to **Read and Write** → save → regenerate Access Token and Secret.

### The form loads but API calls fail (network error or no response)
- Confirm the GAS deployment URL ends in `/exec` (not `/dev`).
- Confirm **Who has access** is set to **Anyone** in the deployment settings.
- Confirm the `GAS_URL` GitHub secret matches the URL exactly (no trailing spaces or newlines).
- Open the browser console (F12) and check the fetch error message.

### GitHub Actions fails on "Inject GAS URL" step
- Confirm the `GAS_URL` secret exists in **Settings → Secrets → Actions**.
- Confirm the secret value is the full URL including `https://` and ending in `/exec`.
- Check the Actions log for the exact error from `inject-env.js`.

### GitHub Pages shows a 404
- Confirm the `gh-pages` branch exists in your repo (created automatically after the first successful Actions run).
- Confirm Pages is configured to deploy from the `gh-pages` branch, root (`/`).
- Wait 1–2 minutes after a deploy — GitHub Pages propagation can take a moment.

### Cron expression does not trigger
- Confirm the `setupTriggers()` function has been run and a `runScheduler` trigger appears in the Triggers view.
- Confirm column C (status) is not already `sent` — the scheduler skips sent rows.
- Check the timezone in `appsscript.json` matches your intended schedule timezone.
- The scheduler fires every minute but only posts when the current minute exactly matches the expression. `0 9 * * 1` posts at precisely 9:00 AM — not 9:01 AM.

### Authorization errors on first run
Click **Review permissions → Advanced → Go to [project name] (unsafe) → Allow**. This is expected for any new unverified Apps Script project.
